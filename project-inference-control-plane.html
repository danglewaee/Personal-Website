Inference Control Plane

289.8 ms p95 90-request mixed-priority benchmark

Context

This project comes from a simple operational pain: serving a model is not the same thing as serving the right model on the right capacity at the right moment. Once traffic, latency, and backend health start moving at the same time, static routing stops being enough.

Why it gets hard

In multi-model inference, one backend can get hot while another stays idle, the first request to a colder model can be slower, and retries can make overload even worse if the system is already under stress.

What most systems still miss

Basic routing assumes every request looks the same and every healthy backend is equally good. That falls apart once request lengths vary, model health changes, and a bad rollout can affect the whole application at once.

What I built

I built a control plane for self-hosted LLM stacks that routes requests using runtime health and load signals, falls back when a backend becomes unhealthy, and gives operators safer rollout behavior through canary and rollback support.

Why teams can trust it

The value is not just smarter routing. It is that the system exposes the operating state teams need to trust under pressure: health, queue depth, latency, request history, fallback behavior, and observability through Prometheus and Grafana.

Mission link

It carries the same question into AI infrastructure: how do we keep a system reliable enough to trust when conditions change underneath it and someone still needs an answer?
